# Reconfiguration-Aware Direct-Connect AI Cluster using Spatial-and-Wavelength-Selective Switching

Brett George1,\*, Weiyang Wang2,† , Zhenguo Wu1,3, Yuyang Wang1,4, Xiang Meng<sup>1</sup> , Manya Ghobadi<sup>2</sup> , and Keren Bergman<sup>1</sup>

*Department of Electrical Engineering, Columbia University, New York, NY 10027, USA Computer Science and Artificial Intelligence Laboratory, MIT, Cambridge, MA 02139, USA Now with NVIDIA Corporation, Santa Clara, CA 95051, U.S.A. Now with Dept. of Electrical and Computer Engineering, University of Connecticut, Storrs, CT 06269, USA \*bcg2133@columbia.edu, †weiyangw@mit.edu*

Abstract: We present a fully reconfiguration-aware direct-connect AI cluster by integrating a spatial-and-wavelength-selective switch with the Linux network stack. We demonstrate simultaneous network reconfiguration with bulk transfers of 6.4-TBytes and multitenant training of four ResNet-18 instances.

#### 1. Introduction

Large-scale network architectures enabled by silicon photonic (SiP) interconnects have emerged as a promising solution to accelerate the growth of artificial intelligence (AI) workloads. SiP architectures are uniquely suited for AI traffic demands with dynamic bandwidth allocation, leading to improvements in job completion times over typical leaf-spine networks [\[1,](#page-2-0) [2\]](#page-2-1). Spatial-and-wavelength-selective switches (SWSS) have been suggested for realizing these architectures due to their fine-grained tunability of parallel wavelength channels [\[3\]](#page-2-2). However, the system-level advantages of implementing SWSS-based interconnects have yet to be demonstrated. This is largely due to the complexity of remedying the hardware and software interface for reconfigurable networks.

In this work, we present the silicon-photonic-enabled Reconfiguration-Aware Direct-Connect AI Cluster (RAD-ICAL), which integrates our custom micro-ring resonator (MRR)-based SWSS with reconfiguration-aware Linux GPU-servers. We deploy a fully packaged 4×4×4λ SWSS that provides four thermally tuned wavelength channels per spatial port. RADICAL utilizes commodity network interface cards (NICs) and transceivers to interface directly with the SWSS for bi-directional connectivity at 25 Gbps per direction. In addition, we implement, to the best of our knowledge, the first Linux kernel packet scheduler for direct-connect reconfigurable networks. We demonstrate network applications including bulk transfer of 6.4 TBytes at line-rate with only a single packet retransmission and multitenant Deep Neural Network (DNN) training of four ResNet-18 instances to 90% accuracy.

Prior work has proposed hybrid network topologies that utilize a combination of electronic packet switches (EPS) and optical circuit switches (OCS) to provide inter-cluster connectivity to the compute units (CU) [\[4,](#page-2-3) [5\]](#page-2-4). Fig. [1\(](#page-0-0)a) shows an example of these architectures that utilize EPSes as the core switches and OCSes, specifically MEMS switches, for the spine of the network. While the OCS-based spines offer network reconfigurability, these systems are bottlenecked by the bandwidth of the EPSes. Previous attempts at addressing link bandwidth constraints have resulted in either highly specialized switches and FPGA NICs [\[6\]](#page-2-5) or the development of custom control planes to handle hybrid switch topologies [\[7\]](#page-2-6). In both cases, the host network interface shown in Fig. [1\(](#page-0-0)b) has proven to be challenging for building a reconfigurable network. Direct-connect network topologies have been put forward as a solution to this problem by implementing a flat network with a single high-bandwidth OCS layer [\[8\]](#page-2-7). RADICAL implements a direct-connect topology shown in Fig. [1\(](#page-0-0)c) and solves the host network interface through a full hardware and software integration using an SWSS and a modified host network stack.

#### 3. Hardware & Software Integration

RADICAL is comprised of four Dell PowerEdge R740xd servers, each of which has an NVIDIA Tesla M40 GPU, an HPE 620QSFP28 NIC, and controller server for a unified control plane as shown in Fig. [2\(](#page-1-0)a). Each server utilizes an SFP28 25 Gbps C-band transceiver for a total of four bi-directional links operating at one of the following dense wavelength division multiplexing (DWDM) ITU channels: C38, C40, C42, and C44. We also use QSFP to SFP converters to interface the transceivers with the NICs. Both the transmit and receive links for each server are connected to an SWSS for network reconfigurability. A diagram for a single bi-directional link is shown in Fig. [2\(](#page-1-0)b). The reconfiguration latency distribution (Fig. [2\(](#page-1-0)c)) for 150 consecutive network reconfigurations ranges from 315.9 ms to 379.6 ms. The main contribution to the latency is from the transceiver locking overhead and the serial communication interface rather than the MRR-based switching elements, which have a thermo-optic time constant on the order of 10−<sup>5</sup> s. For the experiments in the following section, we reconfigure RADICAL between the following two link topologies as shown in Fig. [2\(](#page-1-0)d): (1) GPU 1–GPU 2, GPU 3–GPU 4, and (2) GPU 1–GPU 4, GPU 2–GPU 3. We reconfigure the network by sending voltage control signals via Qontrol Q8iv modules to thermally tune a fully packaged 4×4×4λ switch as shown in Fig. [2\(](#page-1-0)e).

We implement the Linux kernel packet scheduler for direct-connect reconfigurable networks as shown in Fig. [2\(](#page-1-0)f). The host interface relies on two in-kernel modifications: (1) a reconfiguration-aware queuing discipline (qdisc) that briefly holds packets whose destinations are not connected or while the network is reconfiguring, built on the Linux traffic-control framework, and (2) ICMP-based notifier that accepts notification from the switch or its out-of-band manager of an imminent reconfiguration. We maintain state variables to track the currently active destination (curr dst) and whether the interface is under reconfiguration (iface up). This prevents packets from being transmitted during a network blackout. We then modified the NIC's driver code to disable the administrative link-down state to keep the routing information across reconfiguration.

We develop a new type of ICMP message for notifying reconfiguration start and finish, as well as destination change. In conjunction, we implement a unified controller that jointly orchestrates switch reconfiguration and host-side notification. Upon a reconfiguration, this controller pushes pre-tuned voltage control values to thermally tune the SWSS (switch controller). It simultaneously casts ICMP reconfiguration messages to the affected hosts (ICMP controller), triggering the kernel paths above. We note that this kernel interface is compatible with any reconfigurable direct-connect networks.

#### 4. System Evaluation

We demonstrate the software–hardware integration with two end-to-end experiments. First, we run eight iPerf instances (four bi-directional pairs across the two configurations in Fig. [2\(](#page-1-0)d)) while reconfiguring the network automatically every 30 seconds. In total, the network transmitted 6.4-TBytes of data in 10 minutes with only a single packet retransmission. The cumulative per-flow traffic on Fig. [3\(](#page-2-8)a)-(d) shows each iPerf instance alternately saturates the link at 25 Gbps through the SWSS fabric. The scheduler withholds packets toward the inactive destination, demonstrating stable line-rate service across reconfigurations.

![](_page_2_Figure_3.jpeg)

<span id="page-2-8"></span>Fig. 3. (a)-(d) Cumulative data transferred of iPerf instances for four hosts illustrating reconfiguration and line-rate (25 Gbps) communication. (e) Multitenant training setup. (f) Time-to-accuracy plot of four ResNet-18 models converging concurrently. (g)-(h) Training loss and iteration count on GPU 1 and 2, each showing alternating convergence.

We also performed multitenant training with four ResNet-18 jobs under the same network reconfiguration schedule. Each job employs data parallelism over two GPUs that share a link in one configuration, as shown in Fig. [3\(](#page-2-8)e); at any instant, the two jobs with direct paths proceed, while the other two pause due to the packet scheduler withholding their synchronization traffic. The training lasted 100 minutes, using CIFAR-10 for 100 epochs with a local batch size of 1024. All four instances reach 90% accuracy within 45 minutes, as illustrated by Fig. [3\(](#page-2-8)f), confirming concurrent convergence. Fig. [3\(](#page-2-8)g)-(h) zoom into the first 10 minutes of the training on GPUs 1–2, plotting per-iteration loss and iteration counts. The alternative loss reduction and iteration increment illustrate the rotation of link topologies. Collectively, these experiments provide, to our knowledge, the first real-system evidence that an SWSS-based direct-connect reconfigurable fabric sustains line-rate throughput and supports multitenant training to convergence.

## 5. Conclusion

We build a reconfiguration-aware direct-connect AI cluster that integrates a 4×4×4λ SWSS with a Linux packet scheduler. We demonstrate bulk data transfers of 6.4-TBytes over 10 minutes, and multitenant training of four ResNet-18 instances.
